# Техническое задание (ТЗ)

## 1. Общая информация

### 1.1 Наименование проекта
`CollabCode AI` — веб-платформа для совместной разработки кода в комнатах с ИИ-поддержкой и управляемым runtime-контейнером.

### 1.2 Цель проекта
Создать понятную для пользователей и жюри платформу, где команды могут:
- совместно редактировать код в реальном времени;
- запускать код в изолированном окружении;
- получать ИИ-подсказки;
- управлять комнатами, ролями и доступом через личный кабинет.

### 1.3 Целевая аудитория
- студенты и команды хакатонов;
- менторы/жюри для оценки командной активности;
- владельцы комнат (организаторы рабочих сессий).

## 2. Область реализации

### 2.1 Платформенный контур
- Аутентификация пользователей (`register`, `login`, `me`).
- Кабинет пользователя: создание/запуск/остановка комнат.
- Управление приватностью: открытые/закрытые комнаты.
- Управление доступом: роли `owner`, `editor`, `viewer`.

### 2.2 Коллаборация
- Совместное редактирование через Yjs + WebSocket.
- Лента событий комнаты.
- Состояния участников (online/offline).

### 2.3 Runtime и исполнение
- Изоляция runtime комнаты в Docker-контейнере.
- Ручной старт/стоп runtime владельцем.
- Автоостановка runtime при неактивности (idle policy).

### 2.4 Главная страница и аналитика
- Реалтайм-метрики по выбранной комнате.
- Явный выбор источника данных (комната).
- Визуальные блоки и 3D-индикаторы, связанные с backend-данными.

## 3. Функциональные требования

### 3.1 Пользователи и доступ
1. Пользователь может зарегистрироваться по `email + password + name`.
2. Пользователь может войти и получить токен авторизации.
3. Неавторизованный пользователь не должен получать доступ к защищённым API.

### 3.2 Комнаты
1. Пользователь может создать комнату с параметрами:
   - `title`;
   - `goal`;
   - `visibility` (`open`/`closed`);
   - `accessCode` (обязательно для `closed`).
2. Пользователь может войти в комнату:
   - по `roomId` для открытой комнаты;
   - по `roomId + accessCode` для закрытой.
3. Должен действовать лимит одновременных онлайн-участников: до 8.

### 3.3 Роли и права
1. `owner`:
   - управляет runtime комнаты;
   - редактирует цель комнаты;
   - назначает роли участникам (`editor`/`viewer`).
2. `editor`:
   - может редактировать код;
   - может запускать код.
3. `viewer`:
   - режим только чтения;
   - не может редактировать код;
   - не может запускать код.

### 3.4 Инкогнито
1. Пользователь может включать/выключать режим инкогнито в комнате.
2. Для закрытых комнат инкогнито может быть ограничено правилами платформы.

### 3.5 Совместный редактор
1. Все изменения `editor/owner` синхронизируются в реальном времени.
2. Отображаются участники комнаты и их статус.
3. Для `viewer` редактор должен быть в `readOnly`.

### 3.6 ИИ-поддержка
1. После изменений в коде система инициирует ИИ-ревью (debounce).
2. Подсказки отображаются в отдельном блоке с уровнями критичности.
3. События ИИ появляются в ленте событий.

### 3.7 Runtime и контейнеры
1. У каждой комнаты есть собственные runtime-артефакты (`container`, `volume`).
2. При `start`/`stop` статусы runtime отражаются в кабинете.
3. Автоостановка:
   - предупреждение при долгой неактивности;
   - остановка контейнера при превышении порога idle.

### 3.8 Главная страница
1. Данные должны браться из выбранной комнаты.
2. Источник данных должен быть явно показан пользователю.
3. Выбор комнаты должен синхронизироваться с URL (`roomId` в query).

## 4. Нефункциональные требования

1. Язык интерфейса: русский.
2. Поддержка desktop и мобильных экранов.
3. Время обновления метрик на главной: периодический polling (реалтайм-приближенный).
4. Типобезопасность: TypeScript, `typecheck` без ошибок.
5. Безопасность:
   - защищённые API только по Bearer token;
   - проверка прав на уровне backend для критичных действий.

## 5. Архитектура и стек

### 5.1 Frontend
- Next.js (App Router), React, TypeScript
- Monaco Editor, Yjs, WebSocket client
- Framer Motion, Tailwind CSS

### 5.2 Backend
- Node.js + Express + WebSocket (`ws`)
- PostgreSQL для платформенных данных
- Docker orchestration для runtime комнаты

### 5.3 Контейнер исполнения
- `python-runner` image для запуска пользовательского кода

## 6. Модель данных (минимум)

1. `app_users` — пользователи.
2. `room_meta` — метаданные комнаты.
3. `room_memberships` — участники и роли.
4. `room_runtime` — состояние runtime комнаты.
5. `xp_events` — события геймификации (при использовании).

## 8. Пользовательские сценарии

1. Пользователь регистрируется и создаёт комнату.
2. Владелец запускает runtime.
3. Участники входят в комнату, редактируют код и запускают исполнение.
4. Владелец переводит часть участников в `viewer`.
5. `viewer` наблюдает, но не редактирует и не запускает код.
6. На главной странице выбирается комната, и метрики отражают именно её.

## 9. Критерии приёмки

1. Роль `viewer` фактически блокирует редактирование и запуск кода.
2. Управление ролями и участниками работает из кабинета владельца.
3. Закрытая комната требует корректный код доступа.
4. Runtime можно запустить/остановить, статус виден в кабинете.
5. При idle runtime автоматически останавливается по политике.
6. Главная страница показывает источник данных и метрики выбранной комнаты.
7. Проект проходит `npm run typecheck`.

## 10. Ограничения и риски

1. Без серверной проверки прав на уровне WebSocket возможно обходное редактирование кастомным клиентом.
2. При высоком потоке событий нужен контроль шумности event-feed.
3. Для production нужны мониторинг, rate-limit и логирование безопасности.
