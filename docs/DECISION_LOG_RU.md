# 🧠 Обоснование технических решений (Rationale)

Этот документ раскрывает логику выбора технологий и архитектурных подходов в проекте **CollabCode AI**. Наша цель — показать жюри, что за каждым выбором стоит инженерный расчёт, а не просто использование модных инструментов.

---

## 1. Архитектура Монорепозитория (Monorepo)
Мы выбрали структуру монорепозитория (`apps/` + `packages/`), потому что:
- **Типобезопасность (Shared Types)**: Фронтенд и бэкенд используют одни и те же интерфейсы для сообщений через WebSocket. Это исключает 90% ошибок при обновлении протоколов.
- **Единая оркестрация**: Весь проект (БД, API, Клиент) поднимается одной командой `docker-compose up`, что критично для легкого развертывания судьями.

## 2. Совместное редактирование: Yjs (CRDT) против OT
Вместо традиционной Operational Transformation (как в Google Docs), мы использовали **Yjs (Conflict-free Replicated Data Types)**.
- **Почему не OT?** OT требует сложной централизованной логики на сервере, которая плохо масштабируется при больших нагрузках.
- **Преимущество Yjs**: Это распределенный алгоритм. Сервер выполняет лишь роль реле (relay), а вся магия слияния изменений происходит на клиентах. Это обеспечивает мгновенный отклик (отсутствие задержки на подтверждение сервера) и устойчивость к кратковременным обрывам связи.

## 3. Безопасность: Docker Sandboxing для запуска кода
Вместо того чтобы просто выполнять `exec(code)` на сервере, мы используем изолированные **Docker-контейнеры**.
- **Безопасность**: Пользовательский код выполняется в Alpine Linux с минимальным набором прав, без доступа к файловой системе хоста и сети (кроме явно разрешенной).
- **Лимиты ресурсов**: Мы жестко ограничиваем CPU и Memory для каждой «песочницы», защищая сервер от атак типа «бесконечный цикл» или «утечка памяти».
- **Idle Policy**: Контейнеры автоматически останавливаются через 15 минут неактивности, что экономит серверные ресурсы и повышает безопасность.

## 4. Интеграция с ИИ через OpenRouter
Мы не стали привязываться к одному провайдеру (например, только OpenAI), а внедрили **OpenRouter API**.
- **Гибкость**: Жюри может протестировать платформу с любой моделью (GPT-4o, Claude 3.5, Gemini 1.5) просто изменив конфиг, без переписывания бэкенда.
- **Отказоустойчивость**: У нас реализована логика `fallback` — если внешний ИИ недоступен, система переключается на встроенный легковесный анализатор (mock-reviewer), сохраняя работоспособность сессии.

## 5. Фронтенд: Next.js 15 и App Router
- **Скорость загрузки**: Использование Server Components позволяет минимизировать объем JS-бандла, передаваемого клиенту.
- **SEO-готовность**: Даже для внутренней платформы важно иметь правильную структуру заголовков и метаданных для поисковых систем (SEO best practices).
- **DX (Developer Experience)**: Быстрая горячая замена модулей (HMR) ускоряет итерации при разработке новых UI-фич.

## 6. Real-time Уведомления: Telegram и Discord
Мы добавили омниканальность, объединив Telegram и Discord.
- **Кроссплатформенность**: Участники могут получать статус сессии там, где им удобно.
- **Оповещения о событиях**: Уведомления приходят не только при ошибках, но и при важных системных событиях (запуск runtime), что повышает вовлеченность команды.
