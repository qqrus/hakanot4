# Техническое задание (ТЗ) — CollabCode AI
Дата: 2 апреля 2026
Версия: 2.0

## 1. Цель проекта
Создать русскоязычную веб-платформу для совместной разработки в комнатах с:
- коллаборативным редактором кода в реальном времени;
- ролью участников (owner/editor/viewer);
- управляемым runtime-контейнером комнаты;
- AI-ассистентом и AI-навигатором;
- понятной пользовательской аналитикой и дашбордом.

## 2. Целевая аудитория
- студенты и команды хакатонов;
- менторы и жюри;
- владельцы командных рабочих комнат.

## 3. Функциональные требования

### 3.1 Аутентификация и кабинет
- регистрация и вход по email/паролю;
- личный кабинет пользователя;
- создание комнат;
- список моих комнат;
- вход в комнату по `roomId` и/или коду доступа.

### 3.2 Комнаты и роли
- типы комнат: `open` и `closed`;
- роли: `owner`, `editor`, `viewer`;
- owner управляет runtime, целями, ролями;
- viewer не может редактировать код и запускать выполнение.

### 3.3 Коллаборация
- синхронное редактирование кода в реальном времени (Yjs/WebSocket);
- статус участников online/offline;
- лента событий комнаты;
- общий терминал выполнения.

### 3.4 Runtime и контейнеры
- отдельный runtime-контейнер для комнаты;
- команды start/stop runtime из кабинета owner;
- автоостановка по idle timeout;
- сохранение данных комнаты в volume между перезапусками.

### 3.5 AI-функции
- AI-review изменений в коде;
- AI-навигатор по цели комнаты;
- fallback в локальный режим при недоступности внешней модели;
- объяснимые статусы AI в UI.

### 3.6 Главная страница и аналитика
- выбор комнаты как явного источника данных;
- графики и карточки метрик в реальном времени;
- 3D-индикаторы, связанные с backend-метриками;
- понятные пользователю подписи метрик и легенды.

### 3.7 Лидерборд и геймификация
- начисление XP за активность;
- таблица лидерборда;
- выбор комнаты для расчета лидерборда;
- расширение до наград/достижений (этап 4).

### 3.8 Интеграции
- Telegram-уведомления;
- Discord-уведомления;
- тестовый endpoint для проверки интеграций.

## 4. Нефункциональные требования
- интерфейс и тексты: русский язык;
- адаптивность: desktop + mobile;
- запуск всего проекта одной Docker-сборкой;
- типобезопасность (TypeScript), `npm run typecheck` без ошибок;
- проверка прав на backend для критичных действий;
- стабильная работа WebSocket при переподключениях.

## 5. Архитектура (высокоуровнево)
- Frontend: Next.js + React + TypeScript + Tailwind + Framer Motion.
- Backend: Node.js + Express + WebSocket (`ws`).
- Коллаборация: Yjs.
- Хранение: PostgreSQL.
- Runtime: Docker (пер-комнатный lifecycle + volumes).
- Интеграции: OpenRouter API, Telegram Bot API, Discord Webhook.

## 6. API (ключевые группы)
- Auth: `/api/auth/*`
- Rooms/Cabinet: `/api/platform/rooms/*`
- Runtime: `/api/platform/rooms/:roomId/runtime/*`
- Leaderboard: `/api/platform/leaderboard`
- AI Navigator: `/api/platform/rooms/:roomId/navigator`
- Integrations test: `/api/platform/integrations/test`
- Home metrics: `/api/home/:roomId`

## 7. Данные и сущности
- `app_users`
- `room_meta`
- `room_memberships`
- `room_runtime`
- `xp_events`

## 8. Roadmap и статус этапов

### Этап 1: Auth + кабинет + мета-комнаты + роли
Статус: выполнен.
Результат: есть регистрация/вход, кабинет, комнаты, роли и управление участниками.

### Этап 2: Orchestrator + container lifecycle + volume persistence + idle timeout
Статус: выполнен.
Результат: runtime lifecycle, start/stop, сохранение runtime-данных и idle auto-stop.

### Этап 3: Интеграция редактора/AI с новой моделью комнат
Статус: выполнен (с доработками по стабильности).
Результат: связка ролей, коллаборации, AI-статусов, room-context и UI.

### Этап 4: Геймификация и лидерборд
Статус: частично выполнен.
Результат: базовый лидерборд и XP есть; требуется расширение наград и игровых механик.

### Этап 5 (P1): AI-навигатор + Telegram/Discord
Статус: частично выполнен.
Результат: AI-навигатор и интеграционные каналы реализованы; требуется финальная продуктовая полировка.

## 9. Критерии приемки
- viewer не может править код и запускать выполнение;
- owner управляет runtime и ролями без обходов через клиент;
- метрики главной страницы считаются по явно выбранной комнате;
- лидерборд показывает, по какой комнате считается рейтинг;
- Telegram/Discord тестируются через API и доставляют уведомления;
- проект запускается через Docker без ручного старта серверов по отдельности.

## 10. Риски и ограничения
- для free-моделей OpenRouter возможны rate-limit/unstable latency;
- для runtime с `input()` требуется отдельный интерактивный режим (в текущем sandbox ограничен);
- важно исключить утечки секретов и регулярно ротировать API-ключи.
